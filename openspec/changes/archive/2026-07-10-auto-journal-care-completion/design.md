## Context

The care schedule and plant journal currently operate independently. When a user marks a task as done via `CareScheduleUsecases.completeTask()`, a `TaskCompletion` is persisted in the care schedule's own storage (`care_task_completions_v1`), but nothing flows into the plant journal. Both systems use `shared_preferences` with JSON serialization, and both are already registered in GetIt as lazy singletons.

The `JournalEntryType` enum already includes a `task` variant, meaning the journal entity model anticipated this integration â€” it just hasn't been wired yet.

## Goals / Non-Goals

**Goals:**
- When a user marks a care task as complete ("Mark done"), a `JournalEntry` of type `task` is auto-created in the plant journal for that plant.
- The journal entry is indistinguishable from a manually-created task entry in the timeline.
- Snoozing and skipping tasks do not create journal entries.
- Failure to create the journal entry does not roll back the task completion.

**Non-Goals:**
- No UI changes to either the care schedule or journal pages.
- No deletion of journal entries when a completion is deleted from history.
- No migration of past completions into journal entries.
- No new UI to manually trigger auto-journaling.

## Decisions

### Decision 1: Hook into `completeTask()` in `CareScheduleUsecases`

**Chosen**: Add the journal creation call inside `CareScheduleUsecases.completeTask()`, after `repository.recordCompletion()`.

**Rationale**: `completeTask()` is the single, unambiguous entry point for the "Mark done" action. There's exactly one place to modify. The `CareScheduleUsecases` already has cross-use-case dependencies (`PlantCollectionUsecases`, `RoomProfilesUsecases`), so adding `PlantJournalUseCases` is consistent with the existing pattern.

**Alternatives considered:**
- *Hook at the UI layer (page button handler)* â€” Would require every UI entry point to remember to call both systems. Fragile. Also violates Clean Architecture (business logic leaking into presentation).
- *Hook in the repository* â€” Repositories shouldn't know about other features' domain logic. That's the job of use-cases.
- *Use an event bus / listener pattern* â€” Over-engineered for a single integration point. Could evolve into this if more features need to react to task completions.

### Decision 2: Snooze and skip do not journal

**Chosen**: `snoozeTask()` and `skipTask()` remain unchanged â€” they record a `TaskCompletion` for scheduling purposes but do not create journal entries.

**Rationale**: Snoozing/defers mean "I'm not doing this now" â€” not a care event. Skipping means "reset the timer" â€” also not a care event. Creating journal entries for these would pollute the timeline with non-events.

**Alternative considered:**
- *Journal everything* â€” Simple but wrong. Users looking at a journal timeline would see entries like "Skipped watering" which is anti-informative.

### Decision 3: Graceful degradation on journal failure

**Chosen**: If `plantJournal.addEntry()` throws, the exception is caught and logged via `debugPrint`, and the task completion is NOT rolled back.

**Rationale**: The task completion is the primary operation; the journal entry is a secondary side effect. A failure to record in the journal should not prevent the user from successfully marking a task as done. This follows the principle that side-effect failures should not disrupt primary operations.

### Decision 4: Journal entry construction

The auto-created `JournalEntry` is built as follows:

| Field       | Source                                                    |
|-------------|-----------------------------------------------------------|
| `id`        | Generated by `PlantJournalRepository.addEntry()` (UUIDv4) |
| `plantId`   | `task.plantId` (from the `CareTask` being completed)      |
| `type`      | `JournalEntryType.task`                                   |
| `timestamp` | `DateTime.now()` (same moment as the `TaskCompletion`)    |
| `notes`     | `"{taskType.label} completed"` with optional user note appended |
| `photoPath` | `null` (care tasks carry no photos)                       |

The `notes` string uses a localized template from ARB:
- English: `"Watering completed"` or `"Watering completed â€” gave extra due to heat wave"`
- German: `"GieĂźen abgeschlossen"` or `"GieĂźen abgeschlossen â€” wegen Hitze mehr gegeben"`

### Decision 5: No retroactive deletion

If a user deletes a `TaskCompletion` from the care history (via `deleteCompletion()`), the corresponding journal entry is NOT deleted. Journal entries are immutable records of what happened. Deleting from the schedule history only affects the scheduling engine's anchor calculations.

This keeps the two systems loosely coupled and avoids cascading deletes that could surprise users who use the journal as a canonical timeline.

## Risks / Trade-offs

| Risk | Likelihood | Mitigation |
|------|------------|------------|
| `PlantJournalUseCases.addEntry()` blocks task completion if it throws | Low | Wrapped in try-catch with `debugPrint` logging |
| User sees duplicate entries (manual + auto) for the same event | Low | This is explicit user action â€” if they manually journal after completing a task, they get two entries, which is reasonable |
| Performance: `addEntry()` reads/writes full journal list from `shared_preferences` | Medium | This is the existing pattern for all journal operations. If `shared_preferences` becomes a bottleneck, a proper DB (drift/sqlite) migration would address it holistically across all features |
| Journal entry `notes` string is not yet localized | Low | New ARB keys needed â€” tracked in tasks |
| Deleting a completion doesn't clean up the journal | Low | Documented non-goal. Could be added as a follow-up if users request it |
