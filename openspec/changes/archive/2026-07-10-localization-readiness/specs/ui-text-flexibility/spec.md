## ADDED Requirements

### Requirement: All user-facing text widgets have an overflow strategy

Every `Text` widget that displays localizable content SHALL define an overflow behavior (`TextOverflow.ellipsis`, `TextOverflow.clip`, `TextOverflow.fade`, or `TextOverflow.visible`). Text must not overflow its container without a deliberate strategy.

#### Scenario: Long German button label
- **WHEN** a button label reads "Benachrichtigungseinstellungen" in German (vs. "Notification Settings" in English)
- **THEN** the button SHALL either wrap, use ellipsis, or expand to fit the text

### Requirement: Buttons with localizable text use Flexible or minimum width

Action buttons and pill-shaped controls SHALL NOT use fixed pixel widths when they contain localizable text. They SHALL use `IntrinsicWidth`, `Flexible`, or a dynamically-sized layout that accommodates text expansion.

#### Scenario: Fixed-width button clips German text
- **WHEN** a button with `width: 120` contains text that is 140px wide in German
- **THEN** this SHALL be flagged during code review per the ui-text-flexibility checklist
- **AND** the button SHALL be refactored to use intrinsic width or padding-based sizing

### Requirement: Layout review checklist exists

The project SHALL maintain a code-review checklist for localization readiness that covers: overflow strategies, fixed-width containers, concatenated strings (use placeholders instead), text direction assumptions, date format assumptions, and hardcoded unit symbols.

#### Scenario: Reviewer catches hardcoded unit
- **WHEN** a reviewer sees the string `"25°C"` hardcoded in Dart code (not in ARB)
- **THEN** the review SHALL flag it per the localization checklist
- **AND** the developer SHALL replace it with a call to `TemperatureFormatter.format()`

### Requirement: Hardcoded user-facing strings are prohibited

All user-facing text strings SHALL be defined in ARB files (`assets/l10n/l10n_*.arb`) and accessed via `context.l10n`. No user-facing string SHALL be hardcoded in Dart widget code.

#### Scenario: New feature page adds strings
- **WHEN** a developer adds a new feature page with user-facing text
- **THEN** every user-facing string SHALL be added as a new key in the English ARB file (and mirrored in other ARB files with at minimum a placeholder value)
- **AND** accessed via `context.l10n.someKey`

### Requirement: Generated l10n code is never manually edited

Files under `lib/l10n/` (generated by `fvm flutter gen-l10n`) SHALL NOT be edited by hand. All changes SHALL be made in the ARB source files under `assets/l10n/`.

#### Scenario: ARB round-trip
- **WHEN** a developer edits `assets/l10n/l10n_en.arb` and runs `fvm flutter gen-l10n`
- **THEN** `lib/l10n/l10n_en.dart` SHALL be regenerated with the new string
- **AND** any manual edits to `lib/l10n/l10n_en.dart` SHALL be overwritten
